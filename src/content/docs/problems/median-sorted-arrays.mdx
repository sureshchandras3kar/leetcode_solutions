## Problem Statement

Given two sorted arrays `nums1` and `nums2` of size `m` and `n` respectively, return **the median** of the two sorted arrays.

The overall run time complexity should be `O(log (m+n))`.

<Aside type="tip" title="What to notice first">
- Both input arrays are already **sorted**.
- You need to find the median, not just merge and sort.
- The challenge is to do this in logarithmic time, not linear time.
- The median of an array of length n is either the element at index n/2 (if n is odd) or the average of elements at indices n/2-1 and n/2 (if n is even).
</Aside>

### Examples

| nums1 | nums2 | Output | Explanation |
| --- | --- | --- | --- |
| `[1, 3]` | `[2]` | `2.0` | Merged array: [1, 2, 3]. Median is 2. |
| `[1, 2]` | `[3, 4]` | `2.5` | Merged array: [1, 2, 3, 4]. Median is (2 + 3) / 2 = 2.5 |
| `[]` | `[1]` | `1.0` | Single element |

## Constraints

- `nums1.length == m`
- `nums2.length == n`
- `0 <= m <= 1000`
- `0 <= n <= 1000`
- `0 <= nums1[i], nums2[j] <= 10^6`

<Aside type="note" title="Where this pattern appears">
- **Array merging systems** — Database engines merging sorted result sets from multiple indexes or partitions.
- **Statistical computation** — Real-time analytics processing continuous data streams where median is a key metric.
- **Real-time data processing** — Distributed systems computing running medians across multiple data sources without materializing the full dataset.
- **Quantile estimation** — Financial systems calculating median price, execution time, or latency across transactions and requests.
</Aside>

## Complexity Comparison

| Approach | Time | Space | Best When |
| --- | --- | --- | --- |
| Binary Search | O(log(min(m,n))) | O(1) | Optimal — logarithmic time |

## Approach 1: Binary Search (Recommended)

<ApproachTag tag="recommended" />

Use binary search on the shorter array to find the partition point where the left half equals the right half in total length. This achieves O(log(min(m,n))) time.

- Not handling the case where one array is empty.
- Forgetting to handle the case when the total length is odd vs. even.

## Interview Tips

<Aside type="tip" title="Key points to discuss">
- **Why binary search?** — Because both arrays are sorted, you can binary search on the shorter array to partition both arrays such that the left half and right half have equal length.
- **The partition invariant** — The key insight is: if you correctly partition, the maximum of the left half is less than the minimum of the right half.
- **Why O(log(min(m,n)))?** — You search on the shorter array, avoiding unnecessary comparisons when one array is much smaller.
- **Edge cases:** Empty arrays, odd vs. even total length, arrays of vastly different sizes.
- **Follow-up:** "Can you solve it without extra space?" — Both approaches already use O(1) space.
- **Interview signal:** Binary Search shows strong problem-solving skills; Two-Pointer demonstrates understanding of the merge concept.
</Aside>
