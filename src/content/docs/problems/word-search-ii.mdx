## Problem Statement

Given an `m × n` board of characters and a list of strings `words`, return all words on the board.

Each word must be constructed from letters of sequentially adjacent cells, where adjacent cells are horizontally or vertically neighboring. The same letter cell may not be used more than once in one word.

<Aside type="tip" title="What to notice first">
- Build a Trie from all words for efficient prefix matching.
- Use DFS to explore the board and Trie simultaneously.
- Mark visited cells to avoid reusing them in one path.
- Early termination when no matching prefix exists in the Trie.
</Aside>

### Examples

| Board | Words | Output |
| --- | --- | --- |
| `[["o","a","a","n"],["e","t","a","e"],["i","h","k","r"],["i","f","l","v"]]` | `["oath","pea","eat","rain"]` | `["eat","oath"]` |
| `[["a","b"],["c","d"]]` | `["abcb"]` | `[]` |

## Constraints

- `m == board.length`, `n == board[i].length`
- `1 <= m, n <= 12`
- `board[i][j]` is a lowercase English letter.
- `1 <= words.length <= 3 × 10^4`
- `1 <= words[i].length <= 10`

## Real-World Applications

<Aside type="note" title="Where this pattern appears">
- **Spell checkers** — find words from dictionary on a board of letters.
- **Word games** — validate words in Scrabble or Boggle.
- **DNA sequence matching** — find genetic patterns in sequences.
- **Crossword puzzles** — verify words on a grid.
- **Text editors** — multi-string find-and-replace.
</Aside>

## Complexity Comparison

| Approach | Time | Space |
| --- | --- | --- |
| Recursive DFS | O(n × m × 4^l) | O(w × l) |
| Iterative Backtrack | O(n × m × 4^l) | O(w × l) |

*n,m = board dims, l = max word len, w = word count*

## Which approach should you learn first?

- Pick **Recursive DFS** for simplicity — natural for tree/graph traversal.
- Pick **Iterative Backtrack** if you prefer explicit state management.

## Approach 1: Recursive DFS with Trie

<ApproachTag tag="recommended" />

1. Build a Trie from all words.
2. For each cell on the board, start a DFS.
3. Traverse the board following the Trie structure.
4. Mark visited cells by temporarily changing the board.
5. Restore the board during backtracking.

<ComplexityBadges time="O(n×m×4^l)" space="O(w×l)" timeColor="red" spaceColor="yellow" timeNote="Board × word length" spaceNote="Trie size" />

### Pseudocode

```plaintext
function findWords(board, words):
    root = buildTrie(words)
    result = []
    
    for i, j in board:
        dfs(board, i, j, root, result)
    
    return result

function dfs(board, i, j, node, result):
    char = board[i][j]
    if char not in node.children:
        return
    
    nextNode = node.children[char]
    if nextNode.word exists:
        result.add(nextNode.word)
        nextNode.word = null  // Avoid duplicates
    
    board[i][j] = '#'  // Mark visited
    for each neighbor (ni, nj):
        dfs(board, ni, nj, nextNode, result)
    board[i][j] = char  // Restore
```

## Common mistakes

<Aside type="caution" title="Watch for these pitfalls">
- Forgetting to restore the board during backtracking (DFS approach).
- Reusing a cell multiple times in one word.
- Not checking if the character exists in the Trie before recursing.
- Adding duplicate words to the result (set `word = null` after adding).
- Not handling boundary checks before recursing.
</Aside>

## Interview Tips

<Aside type="tip" title="Key trade-offs to discuss">
- **Trie vs. HashSet:** Trie enables early termination when prefix doesn't match; HashSet requires checking all words.
- **Board modification:** Modifying the board saves space but violates immutability; use visited set if needed.
- **Optimization:** Prune the Trie after each word is found to reduce search space.
- **Follow-up:** How would you handle a very large word list? (Stream processing, lazy Trie building.)
- **Edge cases:** Board with one cell, single-letter words, words not on board, overlapping words.
</Aside>
