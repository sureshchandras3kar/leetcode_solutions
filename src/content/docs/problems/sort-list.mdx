## Problem Statement

Sort a linked list in O(n log n) time and O(1) or O(log n) space complexity. Use merge sort for divide and conquer.

<Aside type="tip" title="What to notice first">
- Merge sort is ideal for linked lists (unlike quicksort which needs random access).
- Find middle using slow/fast pointers, split, recursively sort, then merge.
- Iterative approach uses bottom-up merging starting from size 1.
- No need for auxiliary space in merge if done carefully (though most implementations use temp nodes).
</Aside>

### Examples

| Input | Output |
| --- | --- |
| `4 -> 2 -> 1 -> 3` | `1 -> 2 -> 3 -> 4` |
| `-1 -> 5 -> 0` | `-1 -> 0 -> 5` |

## Constraints

- The number of nodes in the list is in the range `[0, 5 * 10^4]`.
- `-10^5 <= Node.val <= 10^5`

## Real-World Applications

<Aside type="note" title="Where this pattern appears">
- Memory allocation and garbage collection
- LRU cache implementation
- Music playlist and browser history
- Undo/Redo functionality in applications
</Aside>

## Complexity Comparison

| Approach | Time | Space |
| --- | --- | --- |
| Recursive | O(n log n) | O(log n) |
| Iterative | O(n log n) | O(1) |

## Which approach should you learn first?

- Pick **Recursive** for clarity — mirrors typical divide and conquer pattern.
- Pick **Iterative** for minimal space and interview confidence.

## Approach 1: Recursive Merge Sort (Recommended)

<ApproachTag tag="recommended" />

2. Split list into two halves.
3. Recursively sort both halves.

## Common mistakes

<Aside type="caution" title="Watch for these pitfalls">
- Incorrect middle finding (use slow/fast pointers).
- Forgetting to split the list (set prev.next = None before recursing).
- Off-by-one errors in merge boundaries.
- Not handling edge cases (empty list, single node).
- Modifying pointers incorrectly during merge.
</Aside>

## Interview Tips

<Aside type="tip" title="Key trade-offs to discuss">
- **Why merge sort?** O(n log n) guaranteed, works well with linked lists (no random access needed).
- **Recursive vs. iterative:** Recursive is cleaner, iterative saves space.
- **Follow-up:** Can you sort in O(1) space and O(n log n) time? (Iterative approach achieves this.)
- **Edge cases:** Empty list, single node, two nodes, all equal values.
</Aside>
