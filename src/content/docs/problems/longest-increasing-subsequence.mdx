## Problem Statement

Given an integer array `nums`, return the length of the longest strictly increasing subsequence.

A subsequence is a sequence that can be derived from an array by deleting some or no elements without changing the order of the remaining elements.

<Aside type="tip" title="What to notice first">
- A subsequence doesn't need to be contiguous.
- The order of elements must be preserved from the original array.
- "Strictly increasing" means each element must be strictly greater than the previous.
- You only need to return the length, not the subsequence itself.
</Aside>

### Examples

| nums | Output | Explanation |
| --- | --- | --- |
| `[10,9,2,5,3,7,101,18]` | `4` | The longest increasing subsequence is `[2,3,7,101]` with length 4 |
| `[0,1,0,4,4,4,3,2,1]` | `2` | The longest increasing subsequence is `[0,1]` or `[0,4]` with length 2 |
| `[3,1,4,1,5,9,2,6,5,3,5]` | `4` | One possible LIS: `[1,4,5,9]` or `[1,2,5,9]` or `[1,2,3,5]` |

## Constraints

- `1 <= nums.length <= 2500`
- `-10^4 <= nums[i] <= 10^4`

## Real-World Applications

<Aside type="note" title="Where this pattern appears">
- **Stock price analysis** — finding the longest period of consistently rising prices.
- **Data compression** — identifying longest increasing runs in datasets.
- **Seismic data processing** — locating increasing event sequences.
- **Version tracking** — finding longest sequence of version improvements.
</Aside>

## Approach 1: Dynamic Programming O(n²)

<ApproachTag tag="recommended" />

dp[i] = length of the longest increasing subsequence ending at index i.

For each i, check all previous indices j where nums[j] < nums[i], and extend the best LIS found.

<ComplexityBadges time="O(n²)" space="O(n)" timeColor="yellow" spaceColor="yellow" timeNote="Nested loops" spaceNote="DP array" />

### Pseudocode

```plaintext
function lisDP(nums):
    n = len(nums)
    dp = array of size n, all initialized to 1
    
    for i from 1 to n - 1:
        for j from 0 to i - 1:
            if nums[j] < nums[i]:
                dp[i] = max(dp[i], dp[j] + 1)
    
    return max(dp)
```

## Approach 2: Binary Search O(n log n)

<ApproachTag tag="interview" />

Maintain a `tails` array where `tails[i]` is the smallest tail element of all LIS of length i+1.

For each number, use binary search to find its position in `tails`. If the position is at the end, we extend the longest LIS; otherwise, we update the tail to possibly get a better LIS.

<ComplexityBadges time="O(n log n)" space="O(n)" timeColor="green" spaceColor="yellow" timeNote="Binary search per element" spaceNote="Tails array" />

### Key Insight

The tails array maintains the invariant that tails is strictly increasing and sorted. At any point, tails[i] contains the smallest possible tail value for an LIS of length i+1. This allows us to use binary search to efficiently find where a new number fits.

### Pseudocode

```plaintext
function lisBinarySearch(nums):
    tails = empty array
    
    for num in nums:
        pos = binary_search_left(tails, num)
        if pos == len(tails):
            tails.append(num)
        else:
            tails[pos] = num
    
    return len(tails)
```

## Common mistakes

<Aside type="caution" title="Watch for these pitfalls">
- Confusing "subsequence" (non-contiguous, order-preserving) with "subarray" (contiguous).
- Initializing dp array to 0 instead of 1 (every element is a LIS of length 1).
- In binary search approach, using the wrong comparison or update logic.
- Forgetting that binary search requires the tails array to remain strictly increasing.
</Aside>

## Interview Tips

<Aside type="tip" title="Key trade-offs to discuss">
- **DP simplicity vs Binary Search efficiency:** DP is easier to code but slower. Binary Search is faster but requires understanding of the tails invariant.
- **Reconstructing the actual subsequence:** Both approaches can be modified to track the actual indices. The DP version is slightly easier.
- **Why binary search works:** The tails array maintains a sorted, strictly increasing sequence. Inserting a smaller value at a position doesn't break the invariant.
- **Space optimization:** Both use O(n) space. DP stores the full state; binary search stores minimal necessary information.
</Aside>
