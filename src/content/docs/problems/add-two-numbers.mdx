## Problem Statement

You are given two non-empty linked lists representing two non-negative integers. The digits are stored in **reverse order** and each of their nodes contains a single digit.

Add the two numbers and return the sum as a linked list in the **same reverse order**.

<Aside type="tip" title="What to notice first">
- The digits are stored in **reverse order**, so the head of the list is the ones place, the next node is the tens place, etc.
- You must handle **carries** propagating through the list, just like elementary school addition.
- The result list may have more nodes than either input list if a carry propagates beyond the last digit.
- Both lists may have different lengths; treat missing nodes as zero.
</Aside>

### Examples

| l1 | l2 | Output | Explanation |
| --- | --- | --- | --- |
| `[2, 4, 3]` | `[5, 6, 4]` | `[7, 0, 8]` | 342 + 465 = 807 (reversed) |
| `[0]` | `[0]` | `[0]` | 0 + 0 = 0 |
| `[9,9,9,9,9,9,9]` | `[9,9,9,9]` | `[8,9,9,9,0,0,0,1]` | 9999999 + 9999 = 10009998 (reversed) |

## Constraints

- The number of nodes in each linked list is in the range `[1, 100]`.
- `0 <= Node.val <= 9`
- It is guaranteed that the list represents a number that does not have leading zeros.

## Real-World Applications

<Aside type="note" title="Where this pattern appears">
- **Big integer arithmetic** — systems that need to add numbers larger than language-native integer types (Bitcoin addresses, cryptography libraries).
- **Polynomial representation** — coefficients in reverse order (lowest power first) allow direct term-by-term addition.
- **Time-series accumulation** — rolling aggregation of digits from streaming data (sensor readings, signal processing).
- **Arbitrary precision math libraries** — implementing addition for Python's unlimited integers or similar systems.
- **Ledger systems** — combining transaction chains where digits represent atomic units.
</Aside>

## Approach 1: Iterative with Carry (Recommended)

<ApproachTag tag="recommended" />

Walk through both lists simultaneously, adding digits column-by-column and propagating the carry forward.

Use a **dummy node** at the start to simplify list construction. At each step, compute the sum of the two digit values plus any carry from the previous step, extract the new carry, and create a new node for the result digit.

<ComplexityBadges time="O(max(m, n))" space="O(max(m, n))" timeColor="green" spaceColor="green" timeNote="One pass through lists" spaceNote="Output list size" />

### Step-by-step Example

**Input:** `l1 = [2, 4, 3]` (342), `l2 = [5, 6, 4]` (465)

| Step | l1 | l2 | carry | sum | digit | new_carry | result |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 1 | 2 | 5 | 0 | 7 | 7 | 0 | `[7]` |
| 2 | 4 | 6 | 0 | 10 | 0 | 1 | `[7, 0]` |
| 3 | 3 | 4 | 1 | 8 | 8 | 0 | `[7, 0, 8]` |
| Done | - | - | 0 | - | - | - | ✓ |

**Input:** `l1 = [9,9,9,9,9,9,9]` (9999999), `l2 = [9,9,9,9]` (9999)

The carry propagates all the way through: `9999999 + 9999 = 10009998`, which becomes `[8,9,9,9,0,0,0,1]` in reverse.

### Pseudocode

```plaintext
function addTwoNumbers(l1, l2):
    dummy = ListNode(0)
    current = dummy
    carry = 0
    
    while l1 or l2 or carry:
        val1 = l1.val if l1 else 0
        val2 = l2.val if l2 else 0
        
        total = val1 + val2 + carry
        carry = total / 10
        digit = total % 10
        
        current.next = ListNode(digit)
        current = current.next
        
        l1 = l1.next if l1 else null
        l2 = l2.next if l2 else null
    
    return dummy.next
```

## Approach 2: Recursive with Carry

<ApproachTag tag="educational" />

Let the recursion handle the "advance to next digit" logic. At each recursive call, add the current digits plus the carry, extract the new carry, and recursively build the rest of the list.

The base case occurs when both lists are exhausted **and** the carry is zero.

<ComplexityBadges time="O(max(m, n))" space="O(max(m, n))" timeColor="green" spaceColor="yellow" timeNote="One pass through lists" spaceNote="Call stack + output" />

### Step-by-step Example

**Input:** `l1 = [2, 4, 3]`, `l2 = [5, 6, 4]`

The recursion mirrors the iterative solution:
- Call 1: `helper(2, 5, 0)` → sum=7, digit=7, carry=0 → create node(7) and recurse
- Call 2: `helper(4, 6, 0)` → sum=10, digit=0, carry=1 → create node(0) and recurse
- Call 3: `helper(3, 4, 1)` → sum=8, digit=8, carry=0 → create node(8) and recurse
- Call 4: `helper(null, null, 0)` → base case, return null
- Build result bottom-up: `7 → 0 → 8 → null`

The recursion naturally builds the list in the correct order because we construct each node after the recursive call returns.

### Pseudocode

```plaintext
function addTwoNumbers(l1, l2):
    return helper(l1, l2, 0)

function helper(l1, l2, carry):
    if not l1 and not l2 and carry == 0:
        return null
    
    val1 = l1.val if l1 else 0
    val2 = l2.val if l2 else 0
    
    total = val1 + val2 + carry
    digit = total % 10
    newCarry = total / 10
    
    nextL1 = l1.next if l1 else null
    nextL2 = l2.next if l2 else null
    
    nextNode = helper(nextL1, nextL2, newCarry)
    return ListNode(digit, nextNode)
```

## Common Mistakes

<Aside type="caution" title="Watch for these pitfalls">
- **Forgetting to handle the final carry:** If two 9s are added, the carry must propagate beyond both lists.
- **Treating null as zero:** When one list is shorter, you must correctly treat missing nodes as having value 0.
- **Losing the carry between iterations:** The carry must persist and be added to every step, not reset.
- **Incorrect carry extraction:** Use integer division (`/`) for carry and modulo (`%`) for the digit, not the other way around.
- **Creating unnecessary nodes:** Only create a new node when you have a digit to store (including the final carry if one exists).
</Aside>

## Key Insights

<Aside type="note" title="Deeper understanding">
- **Why reverse order?** Storing numbers in reverse order (least significant digit first) is a clever representation that allows **O(1) access** to each digit as you traverse. In normal order, you'd need to know the list length first.
- **The dummy node pattern:** A dummy node with value 0 at the start simplifies code by eliminating the special case for the first node. You always append to `current.next`, never reassign `current` itself.
- **Carry is the key challenge:** The hard part is not adding digits—it is ensuring the carry propagates correctly, especially when one list ends before the other or when the final carry needs an extra node.
- **Linked list traversal invariant:** In both approaches, we maintain the invariant: when either l1 or l2 is null, we treat it as 0. This keeps the logic uniform.
</Aside>

## Interview Tips

<Aside type="tip" title="How to discuss this problem">
- **Why iterative over recursive?** Iterative avoids stack overflow on very long lists (though 100 nodes is safe). It is also faster due to no function call overhead.
- **How does reverse order help?** Mention that it allows direct digit-by-digit processing without preprocessing. Standard order would require reversing first.
- **What about the final carry?** Emphasize that the loop condition includes `carry` so a final carry node is created if needed.
- **Edge cases to mention:** single-node lists, lists of different lengths, all-9 inputs (to show carry propagation), and zero values.
- **Follow-up:** Could you solve this if the lists were in normal (non-reversed) order? You'd need to reverse first, or use a stack to process from the tail.
</Aside>
