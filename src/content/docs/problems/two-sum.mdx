## Problem Statement

Given an array of integers `nums` and an integer `target`, return the **indices** of the two numbers such that they add up to `target`.

You may assume that each input has exactly one solution, and you may not use the same element twice. You can return the answer in any order.

<Aside type="tip" title="What to notice first">
- The answer must return **indices**, not the values themselves.
- You are guaranteed exactly one valid pair, so you can stop as soon as you find it.
- The fastest common solution uses a hash map to remember what you have already seen.
</Aside>

### Examples

| Input | Target | Output | Explanation |
| --- | --- | --- | --- |
| `[2, 7, 11, 15]` | `9` | `[0, 1]` | `nums[0] + nums[1] = 2 + 7 = 9` |
| `[3, 2, 4]` | `6` | `[1, 2]` | `nums[1] + nums[2] = 2 + 4 = 6` |
| `[3, 3]` | `6` | `[0, 1]` | `nums[0] + nums[1] = 3 + 3 = 6` |

## Constraints

- `2 <= nums.length <= 10^4`
- `-10^9 <= nums[i] <= 10^9`
- `-10^9 <= target <= 10^9`
- Exactly one valid answer exists.

## Real-World Applications

<Aside type="note" title="Where this pattern appears">
- **Cache lookup / request deduplication** — given a list of request sizes, find two that together fill a fixed-size buffer exactly.
- **Financial balance validation** — check whether two transactions cancel each other out (sum to zero) during reconciliation.
- **Network packet pairing** — pair incoming and outgoing packets whose sizes sum to a known MTU boundary.
- **Two-player game scoring** — determine whether any two remaining moves together hit a target score.
</Aside>

## Complexity Comparison

| Approach | Time | Space | Returns | Best When |
| --- | --- | --- | --- | --- |
| Brute Force | O(n²) | O(1) | Indices | Array is tiny, no extra memory allowed |
| Hash Map | O(n) | O(n) | Indices | General case — fast lookup, unsorted input |
| Two-Pointer | O(n log n) | O(n)\* | Values | Array is already sorted or indices not required |

\* A sorted copy of the array is made to preserve the original order.

## Which approach should you learn first?

- Pick **Hash Map** if you want the best all-around interview answer.
- Pick **Brute Force** if you are still learning how to reason about pairs.
- Pick **Two-Pointer** only if you are practicing the pattern or if the data is already sorted.

## Approach 1: Hash Map (Recommended)

<ApproachTag tag="recommended" />

For each number, compute its complement (`target - num`) and check if that complement was already seen. Store each `number → index` mapping as we iterate.

The key idea is simple: instead of searching the whole array again, remember each number as you go. When the matching partner appears later, you can answer immediately.

<ComplexityBadges time="O(n)" space="O(n)" timeColor="green" spaceColor="yellow" timeNote="Single pass" spaceNote="Hash map storage" />

### Step-by-step Example

**Input:** `nums = [2, 7, 11, 15]`, `target = 9`

| Step | i | num | complement | seen (before) | Action |
| --- | --- | --- | --- | --- | --- |
| 1 | 0 | 2 | 7 | `{}` | 7 not in seen → store `{2: 0}` |
| 2 | 1 | 7 | 2 | `{2: 0}` | 2 found at index 0 → return `[0, 1]` ✓ |

### Pseudocode

```plaintext
function twoSumHashMap(nums, target):
    seen = {}
    for i, num in enumerate(nums):
        complement = target - num
        if complement in seen:
            return [seen[complement], i]
        seen[num] = i
```

## Approach 2: Two-Pointer

<ApproachTag tag="interview" />

Sort the numbers, place one pointer at the beginning and one at the end, then adjust them based on whether the current sum is too small or too large.

<Aside type="caution" title="Returns values, not indices">
Sorting destroys original index positions. This approach returns the **pair values**. Use the Hash Map approach when the original indices are required.
</Aside>

<ComplexityBadges time="O(n log n)" space="O(n)" timeColor="yellow" spaceColor="yellow" timeNote="Sort-dominated" spaceNote="Sorted copy" />

### Step-by-step Example

**Input:** `nums = [3, 2, 4]`, `target = 6`

After sorting: `[2, 3, 4]`

| Step | left | right | sorted[left] | sorted[right] | Sum | Action |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | 0 | 2 | 2 | 4 | 6 | == target → return `[2, 4]` ✓ |

**Input:** `nums = [2, 7, 11, 15]`, `target = 9`

After sorting: `[2, 7, 11, 15]`

| Step | left | right | sorted[left] | sorted[right] | Sum | Action |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | 0 | 3 | 2 | 15 | 17 | > target → right-- |
| 2 | 0 | 2 | 2 | 11 | 13 | > target → right-- |
| 3 | 0 | 1 | 2 | 7 | 9 | == target → return `[2, 7]` ✓ |

### Pseudocode

```plaintext
function twoSumTwoPointer(nums, target):
    sorted = sort(copy of nums)
    left, right = 0, len(sorted) - 1
    while left < right:
        currentSum = sorted[left] + sorted[right]
        if currentSum == target:
            return [sorted[left], sorted[right]]
        elif currentSum < target:
            left++
        else:
            right--
    return []
```

## Approach 3: Brute Force

<ApproachTag tag="simple" />

Check every possible pair using two nested loops.

This is the most direct approach. It is slow for large inputs, but it is a good mental starting point because there is almost no trick to it.

<ComplexityBadges time="O(n²)" space="O(1)" timeColor="red" spaceColor="green" timeNote="All pairs checked" spaceNote="No extra memory" />

### Step-by-step Example

**Input:** `nums = [2, 7, 11, 15]`, `target = 9`

| i | j | nums[i] | nums[j] | Sum | Match? |
| --- | --- | --- | --- | --- | --- |
| 0 | 1 | 2 | 7 | 9 | ✓ → return `[0, 1]` |

**Input:** `nums = [3, 2, 4]`, `target = 6`

| i | j | nums[i] | nums[j] | Sum | Match? |
| --- | --- | --- | --- | --- | --- |
| 0 | 1 | 3 | 2 | 5 | ✗ |
| 0 | 2 | 3 | 4 | 7 | ✗ |
| 1 | 2 | 2 | 4 | 6 | ✓ → return `[1, 2]` |

### Pseudocode

```plaintext
function twoSumBruteForce(nums, target):
    for i in range(len(nums)):
        for j in range(i + 1, len(nums)):
            if nums[i] + nums[j] == target:
                return [i, j]
```

## Common mistakes

<Aside type="caution" title="Watch for these pitfalls">
- Returning the numbers instead of their indices in the hash map solution.
- Reusing the same element twice when the complement equals the current number.
- Forgetting that the two-pointer version here returns values after sorting, not original indices.
</Aside>

## Interview Tips

<Aside type="tip" title="Key trade-offs to discuss">
- **Why hash map over brute force?** Trading O(n) space to drop from O(n²) to O(n) time. Almost always the right call unless memory is severely constrained.
- **Why not two-pointer for the standard problem?** It loses original indices. An interviewer asking for indices is a deliberate signal to use a hash map.
- **Edge cases to mention:** negative numbers (all three approaches handle them), duplicate values like `[3, 3]` (hash map handles this because we check *before* inserting), and the constraint guarantees exactly one solution.
- **Follow-up — Three Sum:** The two-pointer technique extends naturally to three numbers in O(n²).
</Aside>
