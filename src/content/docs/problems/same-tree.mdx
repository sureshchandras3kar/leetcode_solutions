## Problem Statement

Given the roots of two binary trees `p` and `q`, write a function to check if they are the **same**.

Two binary trees are considered the same if they are structurally identical and the nodes have the same values.

<Aside type="tip" title="What to notice first">
- **Structural equality** means both left and right children must exist in both trees (or be absent in both).
- **Value equality** means every node at the same position must have the same integer value.
- An empty tree (null root) is considered the same as another empty tree.
- The test checks both structure AND values simultaneously — one mismatch anywhere fails the comparison.
</Aside>

### Examples

| p | q | Output | Explanation |
| --- | --- | --- | --- |
| `[1,2,3]` | `[1,2,3]` | `true` | Trees are identical in structure and values |
| `[1,2]` | `[1,null,2]` | `false` | Different structure (left vs. right child) |
| `[1,2,1]` | `[1,1,2]` | `false` | Same structure but different values at positions |

## Constraints

- The number of nodes in each tree is in the range `[0, 100]`.
- `-10^4 <= Node.val <= 10^4`

## Real-World Applications

<Aside type="note" title="Where this pattern appears">
- **Tree cloning validation** — confirm that a copied tree structure is identical to the original before applying transformations.
- **Configuration file comparison** — compare two JSON/XML tree structures to detect configuration drift.
- **DOM tree reconciliation** — React and other frameworks compare virtual DOM trees to determine minimal updates.
- **Expression tree equality** — verify that two algebraic expressions represent the same mathematical formula.
- **Backup verification** — ensure a restored directory tree matches the original backup.
</Aside>

## Complexity Comparison

| Approach | Time | Space | Best When |
| --- | --- | --- | --- |
| DFS Recursive | O(min(m, n)) | O(min(h1, h2)) | Small trees; clean recursive code preferred |
| BFS Level-Order | O(min(m, n)) | O(min(w1, w2)) | Wide, shallow trees; iterative approach preferred |

*m and n = number of nodes; h = height; w = max width at any level*

## Which approach should you learn first?

- Learn **DFS Recursive** first — it is elegant, concise, and mirrors the recursive tree structure naturally.
- Use **BFS** as a variation if you want to practice iterative techniques or need explicit level-order thinking.

## Approach 1: DFS Recursive

<ApproachTag tag="recommended" />

Compare nodes recursively by checking:
1. Are both nodes null? (equal)
2. Is exactly one null? (not equal)
3. Do values differ? (not equal)
4. Recursively check left and right subtrees

This mirrors the natural recursive structure of trees and reaches all nodes in minimal time.

<ComplexityBadges time="O(min(m, n))" space="O(min(h1, h2))" timeColor="green" spaceColor="green" timeNote="Each node visited at most once" spaceNote="Call stack depth = tree height" />

### Step-by-step Example

**Input:** `p = [1,2,3]`, `q = [1,2,3]`

```
    1               1
   / \             / \
  2   3           2   3
```

| Step | Check | Result |
| --- | --- | --- |
| Compare roots | `p.val == q.val` (1 == 1) | Continue |
| Left subtree | `isSameTree(p.left=2, q.left=2)` | true |
| Right subtree | `isSameTree(p.right=3, q.right=3)` | true |
| **Result** | Both children match | `true` |

### Pseudocode

```plaintext
function isSameTree(p, q):
    if p is null and q is null:
        return true
    if p is null or q is null:
        return false
    if p.val != q.val:
        return false
    return isSameTree(p.left, q.left) and isSameTree(p.right, q.right)
```

## Approach 2: BFS Level-Order Traversal

<ApproachTag tag="alternative" />

Use a queue to store pairs of nodes from both trees. At each step, dequeue a pair, check if they match, and enqueue the children. This allows you to compare the trees level-by-level (breadth-first) instead of depth-first.

<ComplexityBadges time="O(min(m, n))" space="O(min(w1, w2))" timeColor="green" spaceColor="yellow" timeNote="Each node visited once" spaceNote="Queue holds at most max width nodes" />

### Step-by-step Example

**Input:** `p = [1,2,3]`, `q = [1,2,3]`

| Queue State | Check | Enqueue | Status |
| --- | --- | --- | --- |
| `[(1,1)]` | `1 == 1` | `(2,2)`, `(3,3)` | Match |
| `[(2,2), (3,3)]` | `2 == 2` | `(null,null)`, `(null,null)` | Match |
| `[(null,null), (3,3)]` | Both null | — | Skip |
| `[(3,3)]` | `3 == 3` | `(null,null)`, `(null,null)` | Match |
| `[(null,null), (null,null)]` | Both null | — | Skip |
| `[]` | **All compared** | — | `true` |

### Pseudocode

```plaintext
function isSameTree(p, q):
    queue = [(p, q)]
    while queue is not empty:
        node1, node2 = queue.dequeue()
        if node1 is null and node2 is null:
            continue
        if node1 is null or node2 is null or node1.val != node2.val:
            return false
        queue.enqueue((node1.left, node2.left))
        queue.enqueue((node1.right, node2.right))
    return true
```

<Aside type="caution" title="Watch for these pitfalls">
- **Null handling:** Ensure both null nodes and non-null mismatches are checked before comparing values.
- **Short-circuit evaluation:** Both approaches stop and return `false` as soon as a mismatch is found — efficient pruning.
- **Order doesn't affect correctness:** DFS and BFS visit nodes in different orders but both correctly determine equality.
- **One structure, two algorithms:** The same comparison logic applies; the difference is traversal order.
</Aside>

## Interview Tips

<Aside type="tip" title="Key trade-offs to discuss">
- **DFS vs. BFS:** DFS uses less queue memory for balanced trees but needs call stack space. BFS is iterative and predictable.
- **Early termination:** Both approaches stop on first mismatch — no need to finish traversing the entire trees if inequality is found early.
- **Null nodes matter:** In some problems, two null nodes are considered "the same." Here they explicitly are — be clear on this.
- **Follow-up:** "How would you modify this to check if p is a subtree of q?" (Problem #572) — great discussion point.
- **Tree validation:** This pattern is foundational for tree comparison, cloning verification, and DOM reconciliation in frameworks.
</Aside>
