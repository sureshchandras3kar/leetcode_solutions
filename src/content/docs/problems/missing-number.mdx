## Problem Statement

Given an array `nums` containing `n` distinct numbers in the range `[0, n]`, return the **only number** in that range that is missing from the array.

<Aside type="tip" title="What to notice first">
- The array has exactly `n` elements but covers a range of `n+1` values (0 through n), so exactly one is missing.
- The Gauss sum formula computes the expected total for 0..n in one line — subtract the actual sum and the difference is the missing number.
- Distinct elements are guaranteed, so no duplicates complicate the count.
</Aside>

### Examples

| Input | Output | Explanation |
| --- | --- | --- |
| `[3, 0, 1]` | `2` | Range 0–3, `2` is missing |
| `[0, 1]` | `2` | Range 0–2, `2` is missing |
| `[9,6,4,2,3,5,7,0,1]` | `8` | Range 0–9, `8` is missing |

## Constraints

- `n == nums.length`
- `1 <= n <= 10^4`
- `0 <= nums[i] <= n`
- All numbers in `nums` are unique.

## Real-World Applications

<Aside type="note" title="Where this pattern appears">
- **Sequence gap detection** — find the missing invoice number in an accounting sequence.
- **Packet loss detection** — given a list of received packet sequence numbers, identify which one was dropped.
- **Employee ID audit** — confirm which ID in a contiguous range was never assigned.
- **Memory address validation** — check that all expected memory page indices are accounted for.
</Aside>

## Complexity Comparison

| Approach | Time | Space | Best When |
| --- | --- | --- | --- |
| Hash Set | O(n) | O(n) | Quick to write and easy to understand |
| Gauss Sum | O(n) | O(1) | Memory is constrained; elegant one-liner |

## Which approach should you learn first?

- Learn **Gauss Sum** first — it demonstrates a beautiful mathematical insight and is O(1) space.
- Use **Hash Set** as a fallback if you forget the formula; it is correct and the same time complexity.

## Approach 1: Gauss Sum (Math)

<ApproachTag tag="recommended" />

The sum of all integers from 0 to n is given by the Gauss formula: `n * (n + 1) / 2`. Subtract the actual sum of `nums` from this expected total. The difference is precisely the missing number.

<ComplexityBadges time="O(n)" space="O(1)" timeColor="green" spaceColor="green" timeNote="One pass to sum" spaceNote="Two integer variables" />

### Step-by-step Example

**Input:** `[3, 0, 1]`, n = 3

| Step | Calculation | Value |
| --- | --- | --- |
| Expected sum (0+1+2+3) | 3 × 4 / 2 | `6` |
| Actual sum (3+0+1) | 3 + 0 + 1 | `4` |
| Missing | 6 − 4 | **`2`** |

### Pseudocode

```plaintext
function missingNumber(nums):
    n = len(nums)
    expected = n * (n + 1) / 2
    return expected - sum(nums)
```

## Approach 2: Hash Set

<ApproachTag tag="simple" />

Build a set from `nums`, then iterate 0 through n and return the first value that is not in the set.

<ComplexityBadges time="O(n)" space="O(n)" timeColor="green" spaceColor="yellow" timeNote="Two passes" spaceNote="Set stores n elements" />

### Step-by-step Example

**Input:** `[3, 0, 1]`

| Check | In set? |
| --- | --- |
| 0 | Yes |
| 1 | Yes |
| 2 | **No → return `2`** |

### Pseudocode

```plaintext
function missingNumber(nums):
    seen = set(nums)
    for value in range(len(nums) + 1):
        if value not in seen:
            return value
```

<Aside type="caution" title="Watch for these pitfalls">
- Integer overflow: for very large n, `n * (n + 1)` can overflow a 32-bit integer in C++ and Java — use `long` or `long long`.
- The hash set approach adds O(n) space; the sum approach uses only two variables.
- XOR is another O(n)/O(1) approach: XOR every index 0..n with every element — the missing value remains. Worth mentioning as a bonus.
</Aside>

## Interview Tips

<Aside type="tip" title="Key trade-offs to discuss">
- **Why Gauss sum?** It is O(1) space and a single arithmetic expression — demonstrates mathematical thinking.
- **XOR alternative:** XOR every index 0..n with every element. Pairs cancel; the missing number is left. Equally O(1) space — worth knowing for a follow-up.
- **Edge cases:** `n = 1` with `[0]` → missing is 1; `[1]` → missing is 0. Both formulas handle these correctly.
- **Overflow:** always mention the 32-bit overflow risk for `n × (n+1)` when n is near 10^4.
