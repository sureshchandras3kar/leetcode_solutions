## Problem Statement

Given two strings `s` and `t`, return `true` if `t` is an anagram of `s`, and `false` otherwise.

An anagram is a word formed by rearranging all the letters of another word exactly once. Both strings must use the exact same characters with the exact same frequencies.

<Aside type="tip" title="What to notice first">
- Length check first: if `len(s) != len(t)`, immediately return `false`.
- The key insight is that two strings are anagrams when they have **identical character frequency maps**.
- The hash map approach counts characters in O(n) and needs no extra sorting pass.
</Aside>

### Examples

| s | t | Output | Explanation |
| --- | --- | --- | --- |
| `"anagram"` | `"nagaram"` | `true` | Same letters, different order |
| `"rat"` | `"car"` | `false` | `c` appears in `t` but not `s` |
| `"a"` | `"ab"` | `false` | Different lengths |

## Constraints

- `1 <= s.length, t.length <= 5 * 10^4`
- `s` and `t` consist of lowercase English letters only.

## Real-World Applications

<Aside type="note" title="Where this pattern appears">
- **Spell checker / autocomplete** — detect if the user typed an anagram of a known word (e.g. a transposition typo).
- **Plagiarism detection** — check whether two tokenised sentences use the same vocabulary with the same frequency.
- **Cryptogram solving** — verify that a candidate decoded word uses exactly the letters of the cipher word.
- **Password strength** — flag passwords that are simple rearrangements of dictionary words.
</Aside>

## Complexity Comparison

| Approach | Time | Space | Best When |
| --- | --- | --- | --- |
| Hash Map | O(n) | O(1)\* | General case — single pass, fast |
| Sorting | O(n log n) | O(n) | Code simplicity matters more than speed |

\* At most 26 distinct lowercase letters, so the map size is bounded by a constant.

## Which approach should you learn first?

- Pick **Hash Map** for interviews — O(n) with constant extra space is the expected answer.
- Pick **Sorting** when you want the shortest possible implementation or are familiar with sort-based comparisons.

## Approach 1: Hash Map

<ApproachTag tag="recommended" />

Count each character's frequency in `s` with a hash map, then subtract for each character in `t`. If every count returns to zero, the strings are anagrams. An early-exit check on lengths avoids unnecessary work.

<ComplexityBadges time="O(n)" space="O(1)" timeColor="green" spaceColor="green" timeNote="Two passes of length n" spaceNote="At most 26 keys — constant" />

### Step-by-step Example

**Input:** `s = "rat"`, `t = "tar"`

| Phase | char | Map state |
| --- | --- | --- |
| Count s | r | `{r:1}` |
| Count s | a | `{r:1, a:1}` |
| Count s | t | `{r:1, a:1, t:1}` |
| Subtract t | t | `{r:1, a:1, t:0}` |
| Subtract t | a | `{r:1, a:0, t:0}` |
| Subtract t | r | `{r:0, a:0, t:0}` |
| All zero? | — | **Yes → return `true`** |

### Pseudocode

```plaintext
function isAnagram(s, t):
    if len(s) != len(t): return false
    freq = {}
    for char in s: freq[char] = freq.get(char, 0) + 1
    for char in t:
        freq[char] = freq.get(char, 0) - 1
        if freq[char] < 0: return false
    return true
```

## Approach 2: Sorting

<ApproachTag tag="simple" />

Sort both strings independently. If the sorted versions are equal, the original strings are anagrams — they contain the same characters in the same quantities, just in a different order.

<ComplexityBadges time="O(n log n)" space="O(n)" timeColor="yellow" spaceColor="yellow" timeNote="Two sort passes" spaceNote="Sorted char arrays" />

### Step-by-step Example

**Input:** `s = "anagram"`, `t = "nagaram"`

| Step | Value |
| --- | --- |
| sorted(s) | `"aaagmnr"` |
| sorted(t) | `"aaagmnr"` |
| Equal? | **Yes → return `true`** |

**Input:** `s = "rat"`, `t = "car"`

| Step | Value |
| --- | --- |
| sorted(s) | `"art"` |
| sorted(t) | `"acr"` |
| Equal? | **No → return `false`** |

### Pseudocode

```plaintext
function isAnagram(s, t):
    return sorted(s) == sorted(t)
```

<Aside type="caution" title="Watch for these pitfalls">
- Forgetting the length check — if `s` and `t` have different lengths they cannot be anagrams, so check this first.
- Using a map key that doesn't exist — decrement before checking can produce negative counts; always initialise missing keys to 0 first.
- Unicode follow-up: the constraint here guarantees lowercase ASCII, but in a real system you would want to handle Unicode correctly (use code points, not bytes).
</Aside>

## Interview Tips

<Aside type="tip" title="Key trade-offs to discuss">
- **Why hash map over sorting?** O(n) vs O(n log n). The map approach also lets you exit early the moment a character goes negative.
- **Constant space claim:** with only lowercase English letters the map has at most 26 keys — technically O(1) space even though a map is used.
- **Follow-up — Unicode strings?** The same hash map approach works; the bound on map size increases but the algorithm stays O(n).
- **Follow-up — streams?** You cannot sort a stream, so the hash map approach is the only viable one for streaming input.
