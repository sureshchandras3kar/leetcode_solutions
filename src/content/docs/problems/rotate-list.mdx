## Problem Statement

Given the `head` of a linked list, rotate the list to the right by `k` places.

Rotating a list to the right by `k` places means taking the last `k` nodes and moving them to the front of the list, preserving their relative order.

<Aside type="tip" title="What to notice first">
- If `k` is larger than the list length, you only rotate by `k % length` places since the rotation is cyclic.
- A list of length 1 or an empty list rotated any amount remains unchanged.
- The key challenge is identifying where to "cut" the list and reconnect the pieces.
- Both approaches achieve O(n) time but differ in how they locate the cut point.
</Aside>

### Examples

| Input | k | Output | Explanation |
| --- | --- | --- | --- |
| `1 → 2 → 3 → 4 → 5` | 2 | `4 → 5 → 1 → 2 → 3` | Move the last 2 nodes to the front |
| `0 → 1 → 2` | 4 | `2 → 0 → 1` | k % length = 4 % 3 = 1, rotate by 1 |
| `1` | 1 | `1` | Single node; rotation has no effect |
| `1 → 2` | 2 | `1 → 2` | Full rotation by list length; no change |

## Constraints

- The number of nodes in the list is in the range `[0, 100]`.
- `0 <= Node.val <= 100`
- `0 <= k <= 2 * 10^9`

## Real-World Applications

<Aside type="note" title="Where this pattern appears">
- **Queue rotation** — implement circular queue buffer rotation or round-robin scheduling.
- **Message queue shifting** — rotate message priorities by moving recent items to the front.
- **Playlist rotation** — move the last k songs to the beginning in a music streaming app.
- **Data pipeline buffering** — rotate batches of data in a sliding window operation.
- **Event log rotation** — shift recent events to the front of a time-series log.
</Aside>

| Approach | Time | Space | Passes | Best When |

## Approach 1: Break Point

<ApproachTag tag="recommended" />

Find the node at position `(length - k - 1)` and break the list there. The new head is the node that follows this break point. Reconnect the tail of the rotated portion to the original head.

The key insight is understanding that after a right rotation by k, the node that was at position `length - k` becomes the new head.

<ComplexityBadges time="O(n)" space="O(1)" timeColor="green" spaceColor="green" timeNote="Two passes" spaceColor="No extra memory" />

### Step-by-step Example

**Input:** `1 → 2 → 3 → 4 → 5`, `k = 2`

**Step 1:** Find length = 5

**Step 2:** Normalize k = 2 (already less than 5)

**Step 3:** Find break point at position `5 - 2 - 1 = 2` (the node with value 3)

**Step 4:** Break the list:
- Separate: `1 → 2 → 3` (break here) and `4 → 5`
- Set `3.next = null`
- Connect the tail of `4 → 5` (which is 5) back to 1
- Result: `4 → 5 → 1 → 2 → 3`

**Another case:** `0 → 1 → 2`, `k = 4`

**Step 1:** Find length = 3

**Step 2:** Normalize k = 4 % 3 = 1

**Step 3:** Find break point at position `3 - 1 - 1 = 1` (the node with value 1)

**Step 4:** Break and reconnect:
- `0 → 1` and `2`
- `2.next` points to `0`
- Result: `2 → 0 → 1`

### Pseudocode

```plaintext
function rotateListBreakPoint(head, k):
    if head is null or head.next is null or k == 0:
        return head

    // Step 1: Find length
    length = 0
    current = head
    while current != null:
        length++
        current = current.next

    // Step 2: Normalize k
    k = k % length
    if k == 0:
        return head

    // Step 3: Find break point
    current = head
    for i from 0 to length - k - 2:
        current = current.next

    // Step 4: Break and reconnect
    newHead = current.next
    current.next = null
    tail = newHead
    while tail.next != null:
        tail = tail.next
    tail.next = head

    return newHead
```

## Approach 2: Circular

<ApproachTag tag="interview" />

Create a circular list by connecting the tail back to the head, then locate the new head at position `(length - k)` and break the circle there. This approach elegantly leverages the cyclic nature of rotation.

<ComplexityBadges time="O(n)" space="O(1)" timeColor="green" spaceColor="green" timeNote="Find length and walk to break point" spaceColor="No extra memory" />

### Step-by-step Example

**Input:** `1 → 2 → 3 → 4 → 5`, `k = 2`

**Step 1:** Find length = 5, identify tail (node 5)

**Step 2:** Normalize k = 2

**Step 3:** Create circular list: `5.next = 1` → `1 → 2 → 3 → 4 → 5 → 1 → 2 → 3 → ...`

**Step 4:** Walk `length - k = 5 - 2 = 3` steps from head (starting at position 0):
- Step 0: at node 1
- Step 1: at node 2
- Step 2: at node 3

**Step 5:** Break after node 3:
- `new_head = 3.next = 4`
- `3.next = null`
- Result: `4 → 5 → 1 → 2 → 3`

**Another case:** `0 → 1 → 2`, `k = 4`

**Step 1:** Find length = 3, tail = node 2

**Step 2:** Normalize k = 1

**Step 3:** Create circular list: `2.next = 0`

**Step 4:** Walk `3 - 1 = 2` steps from head:
- Step 0: at node 0
- Step 1: at node 1

**Step 5:** Break after node 1:
- `new_head = 1.next = 2`
- `1.next = null`
- Result: `2 → 0 → 1`

### Pseudocode

```plaintext
function rotateListCircular(head, k):
    if head is null or head.next is null or k == 0:
        return head

    // Step 1: Find length and tail
    length = 0
    tail = head
    while tail.next != null:
        length++
        tail = tail.next
    length++

    // Step 2: Normalize k
    k = k % length
    if k == 0:
        return head

    // Step 3: Create circular list
    tail.next = head

    // Step 4: Find new head position
    stepsToWalk = length - k
    current = head
    for i from 0 to stepsToWalk - 2:
        current = current.next

    // Step 5: Break the circle
    newHead = current.next
    current.next = null

    return newHead
```

## Common Mistakes

<Aside type="caution" title="Watch for these pitfalls">
- **Forgetting to normalize k** — if k is larger than the list length, the rotation becomes k % length.
- **Off-by-one errors in finding the break point** — ensure you find the node *before* the rotation point, not the rotation point itself.
- **Not handling edge cases** — single-node lists, empty lists, and k=0 should return the original head.
- **Creating actual circles in the circular approach** — remember to break the circle after finding the new head.
- **Losing the tail reference** — in both approaches, ensure you correctly identify and reconnect the tail.
</Aside>

## Interview Tips

<Aside type="tip" title="Key trade-offs to discuss">
- **Break Point vs. Circular:** Both achieve O(n) time and O(1) space. Break Point is more intuitive (two distinct phases), while Circular is more elegant and shows deeper understanding of linked list properties.
- **Why normalize k?** Emphasize that rotations are cyclic: rotating by k is the same as rotating by k % length. This prevents unnecessary iterations.
- **Why find (length - k - 1)?** Explain that after rotation, the new head is at the old position (length - k). To perform the break, you need the node *before* it.
- **Edge cases to mention:** empty lists, single-node lists, k = 0, k = length, and k > length. All should be handled gracefully.
- **Follow-up — Reverse Sections:** You can combine rotation with reversal to solve more complex linked list problems.
</Aside>
