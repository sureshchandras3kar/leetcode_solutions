## Problem Statement

You are given the heads of two sorted linked lists `list1` and `list2`. Merge the two lists in a **single sorted list**. The list should be made by splicing together the nodes of the two lists.

Return the head of the merged linked list.

<Aside type="tip" title="What to notice first">
- You are working with **linked lists**, not arrays, so you cannot jump to arbitrary indices.
- Both input lists are already **sorted**, so you can decide which node to add next by simple comparison.
- You must use pointer manipulation; you cannot create new nodes (except for a dummy node to simplify code).
- The traversal pattern is similar to merging two sorted arrays, but with node pointers instead of indices.
</Aside>

### Examples

| list1 | list2 | Output | Explanation |
| --- | --- | --- | --- |
| `[1, 2, 4]` | `[1, 3, 4]` | `[1, 1, 2, 3, 4, 4]` | All six nodes merged in sorted order |
| `[]` | `[]` | `[]` | Both lists are empty |
| `[]` | `[0]` | `[0]` | First list is empty, return second list |

## Constraints

- The number of nodes in both lists is in the range `[0, 50]`.
- `-100 <= Node.val <= 100`
- Both `list1` and `list2` are sorted in non-decreasing order.

## Real-World Applications

<Aside type="note" title="Where this pattern appears">
- **File merging** — combining two sorted file streams (log files, time-series data) into a single sorted output.
- **Database operations** — merging results from two sorted database queries or indexes.
- **Merge sort implementation** — the merge step of a divide-and-conquer sorting algorithm.
- **Stream processing** — combining two ordered event streams in real-time analytics.
- **Playlist management** — merging two chronologically sorted playlists into a unified timeline.
</Aside>

## Approach 1: Iterative (Recommended)

<ApproachTag tag="recommended" />

Maintain a pointer to the tail of the merged list and step through both input lists, always appending the smaller node to the result.

The key idea is simple: maintain a tail pointer that points to the last node of the merged list. Compare the next nodes from both lists, append the smaller one, and advance its pointer. When one list is exhausted, append the entire remainder of the other list.

<ComplexityBadges time="O(m + n)" space="O(1)" timeColor="green" spaceColor="green" timeNote="Single pass through both lists" spaceNote="Only pointer variables" />

### Pseudocode

```plaintext
function mergeTwoSortedListsIterative(list1, list2):
    dummy = new ListNode(0)
    current = dummy
    
    while list1 and list2:
        if list1.val <= list2.val:
            current.next = list1
            list1 = list1.next
        else:
            current.next = list2
            list2 = list2.next
        current = current.next
    
    // Append remaining nodes
    if list1:
        current.next = list1
    else:
        current.next = list2
    
    return dummy.next
```

## Approach 2: Recursive

<ApproachTag tag="interview" />

At each step, decide which of the two next nodes (from `list1` or `list2`) should come first, set it to point to the merged result of the remaining lists, and return it.

This approach is elegant because the problem exhibits **optimal substructure**: merging two lists is the same as choosing the smaller head and then merging the remainder. The recursion naturally mirrors the structure of linked lists.

<Aside type="note" title="Recursion vs Iteration tradeoff">
The recursive solution is cleaner conceptually but uses O(m + n) extra space for the call stack. For very long lists, the iterative approach is safer.
</Aside>

<ComplexityBadges time="O(m + n)" space="O(m + n)" timeColor="green" spaceColor="yellow" timeNote="Single pass through both lists" spaceNote="Call stack depth" />

### Step-by-step Example

**Input:** `list1 = [1, 2, 4]`, `list2 = [1, 3, 4]`

The recursion tree shows how the problem breaks down:

```
merge([1, 2, 4], [1, 3, 4])
├─ 1 <= 1 → return 1.next = merge([2, 4], [1, 3, 4])
│  ├─ 2 > 1 → return 1.next = merge([2, 4], [3, 4])
│  │  ├─ 2 < 3 → return 2.next = merge([4], [3, 4])
│  │  │  ├─ 4 > 3 → return 3.next = merge([4], [4])
│  │  │  │  ├─ 4 <= 4 → return 4.next = merge(null, [4])
│  │  │  │  │  └─ return 4
│  │  │  │  └─ result: 4 → 4
│  │  │  └─ result: 3 → 4 → 4
│  │  └─ result: 2 → 3 → 4 → 4
│  └─ result: 1 → 2 → 3 → 4 → 4
└─ result: 1 → 1 → 2 → 3 → 4 → 4
```

### Pseudocode

```plaintext
function mergeTwoSortedListsRecursive(list1, list2):
    // Base cases
    if list1 is null:
        return list2
    if list2 is null:
        return list1
    
    // Recursive case: choose the smaller head and recurse
    if list1.val <= list2.val:
        list1.next = mergeTwoSortedListsRecursive(list1.next, list2)
        return list1
    else:
        list2.next = mergeTwoSortedListsRecursive(list1, list2.next)
        return list2
```

## Common mistakes

<Aside type="caution" title="Watch for these pitfalls">
- **Creating new nodes instead of reusing:** Both input lists' nodes should be reused in the output. Only the dummy node (if used) should be newly created.
- **Forgetting to handle empty lists:** If one or both lists are empty, the result should simply be the other list (or null if both are empty).
- **Not advancing pointers correctly:** Forgetting to move `list1` or `list2` after attaching a node can cause an infinite loop or incorrect merge.
- **Losing reference to the merged list head:** Make sure you return the node after the dummy (i.e., `dummy.next`), not the dummy itself.
- **Recursion stack overflow on very long lists:** For production code, iterative is safer; for interviews, mention this tradeoff.
</Aside>

## Edge cases to handle

| Case | Behavior | Example |
| --- | --- | --- |
| Both lists empty | Return null | `[]` + `[]` → `[]` |
| First list empty | Return second list | `[]` + `[1, 2]` → `[1, 2]` |
| Second list empty | Return first list | `[1, 2]` + `[]` → `[1, 2]` |
| Single-node lists | Merge correctly | `[1]` + `[2]` → `[1, 2]` |
| Duplicate values | Merge in stable order | `[1, 3]` + `[1, 3]` → `[1, 1, 3, 3]` |
| Negative values | Treat as regular integers | `[-2, 0, 2]` + `[-1, 1]` → `[-2, -1, 0, 1, 2]` |

## Interview Tips

<Aside type="tip" title="Key points to discuss">
- **Why iterative over recursive?** The iterative approach uses O(1) extra space (excluding output) versus O(m + n) for the call stack. For interviews, mention both and discuss the tradeoff.
- **Why no new nodes?** You are reusing the input nodes' structure; only the connections change. A dummy node is an exception — it simplifies code by eliminating special cases.
- **Why compare and choose?** Both lists are sorted, so you only need to compare heads. This greedy choice is optimal.
- **Follow-up — Merge k Sorted Lists:** Extend this to merge multiple lists using a priority queue or merge pairwise.
- **Follow-up — In-place merge of arrays:** If you had two sorted arrays, you'd merge in-place from right to left to avoid overwriting unprocessed elements.
- **Edge case emphasis:** Always check empty list handling in interviews; it is a common mistake and easily caught.
</Aside>
