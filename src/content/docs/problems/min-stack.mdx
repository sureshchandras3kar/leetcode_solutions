## Problem Statement

Design a stack that supports `push`, `pop`, `top`, and retrieving the **minimum element** in **constant time**.

Implement the `MinStack` class:

- `MinStack()` Initializes the stack object.
- `void push(int val)` Pushes the element `val` onto the stack.
- `void pop()` Removes the element on the top of the stack.
- `int top()` Gets the top element of the stack.
- `int getMin()` Retrieves the minimum element in the stack.

Each operation must run in **O(1) time**.

<Aside type="tip" title="What to notice first">
- The stack must support all four operations in constant time.
- The key challenge is: how can we track the minimum without scanning the entire stack on every `getMin()` call?
- You cannot modify the values being pushed; you must design the data structure cleverly.
- Every `push` and `pop` must maintain the minimum invariant.
</Aside>

### Examples

| Operation | Stack | Min | Notes |
| --- | --- | --- | --- |
| `push(-2)` | `[-2]` | -2 | Initialize with minimum as -2 |
| `push(0)` | `[-2, 0]` | -2 | New min is still -2 |
| `push(-3)` | `[-2, 0, -3]` | -3 | New min is -3 |
| `getMin()` | `[-2, 0, -3]` | -3 | Return -3 ✓ |
| `pop()` | `[-2, 0]` | -2 | Remove -3, min reverts to -2 |
| `top()` | `[-2, 0]` | — | Return 0 ✓ |
| `getMin()` | `[-2, 0]` | -2 | Return -2 ✓ |

## Constraints

- `-2^31 <= val <= 2^31 - 1`
- `pop`, `top`, and `getMin` operations will always be called on **non-empty stacks**.
- At most `3 × 10^4` calls will be made to `push`, `pop`, `top`, and `getMin`.

## Real-World Applications

<Aside type="note" title="Where this pattern appears">
- **Stock price tracking** — maintain running minimum price alongside incoming prices during trading sessions.
- **Browser history navigation** — track the minimum (or maximum) time spent on each visited page.
- **Network latency monitoring** — retrieve minimum latency within a sliding time window as packets arrive.
- **Undo/redo systems** — maintain a minimum or maximum state value as operations are pushed and popped.
- **Expression evaluation** — some parsing tasks need to track minimum depth or priority level as tokens arrive.
</Aside>

## Approach 1: Dual Stack (Recommended)

<ApproachTag tag="recommended" />

Maintain **two stacks**:
1. **Main stack**: stores all pushed values.
2. **Min stack**: stores the minimum value at each level (only pushes/pops in sync with main stack).

Whenever we push a value to the main stack, we push `min(val, current_min)` to the min stack. When we pop from the main stack, we pop from the min stack too. This way, `getMin()` is always O(1) by peeking the top of the min stack.

<ComplexityBadges time="O(1)" space="O(n)" timeColor="green" spaceColor="yellow" timeNote="All operations constant" spaceNote="Two stacks of size n" />

### Step-by-step Example

**Sequence of operations:** `push(-2)`, `push(0)`, `push(-3)`, `getMin()`, `pop()`, `top()`, `getMin()`

| Step | Operation | Main Stack | Min Stack | Result | Action |
| --- | --- | --- | --- | --- | --- |
| 1 | `push(-2)` | `[-2]` | `[-2]` | — | First element: min is -2 itself |
| 2 | `push(0)` | `[-2, 0]` | `[-2, -2]` | — | min(-2, 0) = -2, so push -2 |
| 3 | `push(-3)` | `[-2, 0, -3]` | `[-2, -2, -3]` | — | min(-2, -3) = -3, so push -3 |
| 4 | `getMin()` | `[-2, 0, -3]` | `[-2, -2, -3]` | **-3** | Peek top of min stack → -3 ✓ |
| 5 | `pop()` | `[-2, 0]` | `[-2, -2]` | — | Remove from both stacks |
| 6 | `top()` | `[-2, 0]` | `[-2, -2]` | **0** | Peek main stack → 0 ✓ |
| 7 | `getMin()` | `[-2, 0]` | `[-2, -2]` | **-2** | Peek min stack → -2 ✓ |

### Pseudocode

```plaintext
class MinStack:
    minStack = {} // main stack for all values
    mainStack = {} // separate stack for minimums

    function push(val):
        mainStack.push(val)
        if minStack is empty:
            minStack.push(val)
        else:
            minStack.push(min(val, minStack.top()))

    function pop():
        mainStack.pop()
        minStack.pop()

    function top():
        return mainStack.top()

    function getMin():
        return minStack.top()
```

## Approach 2: Single Stack with Pairs

<ApproachTag tag="interview" />

Instead of maintaining two separate stacks, store **one element per level as a pair (value, min_at_this_level)**.

Each pair holds:
- **value**: the actual value being pushed
- **min_at_this_level**: the minimum from the bottom of the stack up to and including this element

When you push a new value, compute its minimum as `min(val, previous_min)` and store the pair. This way, the entire history of minimums is embedded in the stack itself.

<Aside type="note" title="Memory efficiency">
Both approaches use O(n) space overall, but the pair approach stores everything in one data structure, which can be slightly more cache-friendly. However, the dual stack approach is often clearer to reason about in interviews.
</Aside>

<ComplexityBadges time="O(1)" space="O(n)" timeColor="green" spaceColor="yellow" timeNote="All operations constant" spaceNote="Single stack with pairs" />

### Step-by-step Example

**Sequence of operations:** `push(-2)`, `push(0)`, `push(-3)`, `getMin()`, `pop()`, `top()`, `getMin()`

| Step | Operation | Stack | Top Pair | Min | Action |
| --- | --- | --- | --- | --- | --- |
| 1 | `push(-2)` | `[(-2, -2)]` | `(-2, -2)` | -2 | First: value=min=-2 |
| 2 | `push(0)` | `[(-2, -2), (0, -2)]` | `(0, -2)` | -2 | min(0, -2) = -2 |
| 3 | `push(-3)` | `[(-2, -2), (0, -2), (-3, -3)]` | `(-3, -3)` | -3 | min(-3, -2) = -3 |
| 4 | `getMin()` | `[(-2, -2), (0, -2), (-3, -3)]` | `(-3, -3)` | **-3** | Return second element of top pair |
| 5 | `pop()` | `[(-2, -2), (0, -2)]` | `(0, -2)` | -2 | Remove top pair |
| 6 | `top()` | `[(-2, -2), (0, -2)]` | `(0, -2)` | — | **Return 0** (first element of top pair) |
| 7 | `getMin()` | `[(-2, -2), (0, -2)]` | `(0, -2)` | **-2** | Return second element of top pair |

### Pseudocode

```plaintext
class MinStack:
    stack = [] // stores (value, min_at_this_level) pairs

    function push(val):
        if stack is empty:
            stack.push((val, val))
        else:
            currentMin = min(val, stack.top().min)
            stack.push((val, currentMin))

    function pop():
        stack.pop()

    function top():
        return stack.top().value

    function getMin():
        return stack.top().min
```

## Common mistakes

<Aside type="caution" title="Watch for these pitfalls">
- **Forgetting to sync the min stack on pop()** — if you only pop from the main stack and not the min stack, the two get out of sync.
- **Computing min inside pop() instead of push()** — the minimum must be stored *as you push*, not computed later.
- **Using a heap or priority queue** — those are O(log n) per operation, not O(1).
- **Recomputing the minimum by scanning the stack on getMin()** — this is O(n), defeating the entire purpose.
- **Not handling duplicate minimums** — if the same minimum value appears multiple times, make sure each occurrence is tracked in the min stack.
- **Off-by-one errors when storing pairs** — ensure the min value in a pair reflects the minimum *up to that point*, not just the current value.
</Aside>

## Interview Tips

<Aside type="tip" title="Key trade-offs to discuss">
- **Dual Stack vs. Pairs:** Both are O(1) time. Dual Stack is clearer and easier to explain. Pairs are more compact but require careful bookkeeping.
- **Why not use a heap?** Heaps give O(log n) per operation, and you'd need to remove specific elements (no efficient removal for min-heaps).
- **Why not recompute on getMin()?** The problem explicitly demands O(1) time for getMin(). Scanning the stack is O(n).
- **Edge cases to mention:** Single element (min = that element), duplicate minimums (each occurrence needs a min stack entry), negative numbers (all approaches handle them), and large stacks (space is O(n) either way).
- **Follow-up — Median Finder:** A harder version where you track both min and median; this requires a heap-based approach.
</Aside>
