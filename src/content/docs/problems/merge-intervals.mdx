## Problem Statement

Given an array of `intervals` where `intervals[i] = [starti, endi]`, merge all overlapping intervals, and return an array of the non-overlapping intervals that cover all the intervals in the input.

<Aside type="tip" title="What to notice first">
- Two intervals `[a, b]` and `[c, d]` overlap if `c <= b`.
- After merging, the result should contain non-overlapping intervals in sorted order.
- A simple sort-and-scan approach is often optimal for this problem.
</Aside>

### Examples

| Input | Output | Explanation |
| --- | --- | --- |
| `[[1,3],[2,6],[8,10],[15,18]]` | `[[1,6],[8,10],[15,18]]` | [1,3] and [2,6] overlap, merge to [1,6]. |
| `[[1,4],[4,5]]` | `[[1,5]]` | [1,4] and [4,5] overlap at the boundary. |
| `[[1,2],[1,2]]` | `[[1,2]]` | Identical intervals merge to one. |

## Constraints

- `1 <= intervals.length <= 10^4`
- `intervals[i].length == 2`
- `0 <= starti <= endi <= 10^4`

<Aside type="note" title="Where this pattern appears">
- **Calendar event merging** — Scheduling systems consolidating overlapping time blocks from multiple calendars or shared resources.
- **Resource allocation** — Server and container orchestration merging time windows of resource availability or maintenance windows.
- **Time slot consolidation** — Educational or booking systems merging adjacent or overlapping appointment slots into contiguous blocks.
- **Meeting scheduling** — Calendar applications automatically merging or detecting conflicts between overlapping meeting times across participants.
</Aside>

## Complexity Comparison

| Approach | Time | Space | Best When |
| --- | --- | --- | --- |
| Sort and Merge | O(n log n) | O(1) | Optimal — simplest and fastest |
| Heap | O(n log n) | O(n) | Alternative structure |

## Which approach should you learn first?

- Pick **Sort and Merge** for the optimal interview solution — it is intuitive and efficient.
- Pick **Heap** if you want to practice heap-based interval handling.

## Approach 1: Sort and Merge (Recommended)

<ApproachTag tag="recommended" />

## Common mistakes

<Aside type="caution" title="Watch for these pitfalls">
- Forgetting to sort intervals before merging.
- Using `<` instead of `<=` when checking for overlaps.
- Not updating the end of the current interval when merging.
- Handling edge cases like adjacent intervals [1,2] and [2,3].
</Aside>

## Interview Tips

<Aside type="tip" title="Key points to discuss">
- **Sort first** — Sorting by start time is the crucial insight that makes merging linear in the sorted pass.
- **Overlap detection** — Two intervals overlap if the next start is within or at the end of the current interval: `next.start <= current.end`.
- **Why Sort and Merge is optimal** — Both approaches are O(n log n) due to sorting, but Sort and Merge uses O(1) space, making it superior.
- **Heap alternative** — If intervals arrive in a stream, a heap lets you process them online without sorting the entire input first.
- **Edge cases:** Adjacent intervals like [1,2] and [2,3] (should merge), duplicate intervals, single interval.
- **Follow-up:** "What if intervals arrive in a stream?" → Use a heap-based approach or balanced tree for dynamic merging.
</Aside>
