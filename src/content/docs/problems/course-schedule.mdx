## Problem Statement

You are given a number of `courses` labeled from `0` to `numCourses - 1` and a list of prerequisite pairs `prerequisites` where `prerequisites[i] = [ai, bi]` means you must take course `bi` before taking course `ai`. Return `true` if you can finish all courses, otherwise return `false`.

<Aside type="tip" title="Key Insight">
- This is a cycle detection problem in a directed graph
- If there's a cycle in the prerequisite graph, it's impossible to complete all courses
- Think about topological sorting — a valid ordering exists only if the graph is acyclic
- Both DFS and BFS (Kahn's algorithm) can detect cycles effectively
</Aside>

### Examples

| numCourses | prerequisites | Output | Explanation |
| --- | --- | --- | --- |
| 2 | `[[1,0]]` | `true` | Take course 0 first, then course 1 |
| 2 | `[[1,0],[0,1]]` | `false` | Circular dependency: course 0 requires 1, course 1 requires 0 |
| 3 | `[[0,1],[0,2],[1,2]]` | `true` | Take course 2, then 1, then 0 |
| 4 | `[[1,0],[2,0],[3,1],[3,2]]` | `true` | Valid topological order: 0 → 1 → 2 → 3 and 0 → 2 → 3 |

## Constraints

- `1 <= numCourses <= 2000`
- `0 <= prerequisites.length <= 5000`
- `prerequisites[i].length == 2`
- `0 <= ai, bi < numCourses`
- All values are unique

## Real-World Applications

<Aside type="note" title="Where this pattern appears">
- **Build system dependency resolution** — checking if build targets can be compiled without circular dependencies
- **Task scheduling** — determining if all tasks can be executed respecting their prerequisites
- **Curriculum planning** — validating course requirements don't have circular dependencies
- **Software module loading** — ensuring module imports don't form circular reference chains
</Aside>
