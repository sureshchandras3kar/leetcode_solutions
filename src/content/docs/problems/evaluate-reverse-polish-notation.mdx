## Problem Statement

Evaluate the value of an arithmetic expression in Reverse Polish Notation (RPN).

Valid operators are `+`, `-`, `*`, `/`. Each operand may be an integer or another expression.

<Aside type="tip" title="What to notice first">
- In RPN, operators come **after** their operands: `3 4 +` means `3 + 4 = 7`
- A **stack** naturally models this: push operands, pop two operands when you see an operator
- Order matters for `-` and `/`: the first pop is the second operand
- Division truncates towards zero (not floor division)
</Aside>

### Examples

| Input | Output | Explanation |
| --- | --- | --- |
| `["2", "1", "+", "3", "*"]` | `9` | (2 + 1) * 3 = 3 * 3 = 9 |
| `["4", "13", "5", "/", "+"]` | `6` | 4 + (13 / 5) = 4 + 2 = 6 |
| `["10", "6", "9", "3", "+", "-11", "*", "/", "*", "17", "+", "5", "+"]` | `22` | Complex nested operations = 22 |
| `["3", "4", "+"]` | `7` | 3 + 4 = 7 |
| `["15"]` | `15` | Single number is valid |

## Constraints

- `1 <= tokens.length <= 10^4`
- Each token is either an operator or a valid integer in range `[-200, 200]`
- The input guarantees a valid RPN expression

## Real-World Applications

<Aside type="note" title="Where this pattern appears">
- **Calculator implementations** — many pocket calculators use RPN to avoid parentheses
- **Compiler expression evaluation** — intermediate code often uses postfix notation
- **Spreadsheet formula engines** — some internal representations use RPN
- **Undo/redo in editors** — operations stored in postfix order for efficient processing
</Aside>

## Approach 1: Stack-Based Evaluation (Recommended)

<ApproachTag tag="recommended" />

Iterate through tokens. Push operands onto the stack. When you see an operator, pop the required operands, compute, and push the result back.

The key insight: the stack automatically handles operator precedence because RPN is **unambiguous** — there are no parentheses to parse.

<ComplexityBadges time="O(n)" space="O(n)" timeColor="green" spaceColor="yellow" timeNote="Single pass through tokens" spaceNote="Stack size proportional to expression depth" />

### Step-by-step Example

**Input:** `["2", "1", "+", "3", "*"]`

| Step | Token | Stack (before) | Action | Stack (after) |
| --- | --- | --- | --- | --- |
| 1 | `"2"` | `[]` | Push 2 | `[2]` |
| 2 | `"1"` | `[2]` | Push 1 | `[2, 1]` |
| 3 | `"+"` | `[2, 1]` | Pop 1, 2 → 2+1=3 | `[3]` |
| 4 | `"3"` | `[3]` | Push 3 | `[3, 3]` |
| 5 | `"*"` | `[3, 3]` | Pop 3, 3 → 3*3=9 | `[9]` |
| Result | — | — | — | `9` |

### Pseudocode

```plaintext
function evaluateRpn(tokens):
    stack = []
    for token in tokens:
        if token is an operator:
            b = stack.pop()
            a = stack.pop()
            result = applyOperator(a, token, b)
            stack.push(result)
        else:
            stack.push(int(token))
    return stack[0]
```

## Approach 2: Recursive Evaluation

<ApproachTag tag="interview" />

Process tokens from right to left. When you encounter an operand, return it. When you encounter an operator, recursively get the two operands and apply the operator.

This mirrors the stack approach but uses the call stack instead of an explicit stack.

<ComplexityBadges time="O(n)" space="O(n)" timeColor="green" spaceColor="yellow" timeNote="Single pass through tokens" spaceNote="Recursion call stack depth" />

### Step-by-step Example

**Input:** `["4", "13", "5", "/", "+"]`

Processing from right to left:
1. See `"+"` → recursively get right operand
2. See `"/"` → recursively get right operand (5), left operand (13) → 13 / 5 = 2
3. Back to `"+"` → left operand is 4 → 4 + 2 = 6

### Pseudocode

```plaintext
function evaluateRpnRecursive(tokens, index):
    token = tokens[index]
    index--
    
    if token is an operator:
        right = evaluateRpnRecursive(tokens, index)
        left = evaluateRpnRecursive(tokens, index)
        return applyOperator(left, token, right)
    else:
        return int(token)
```

## Common Mistakes

<Aside type="caution" title="Watch for these pitfalls">
- **Operand order:** The first pop is the **second** operand. For `5 - 3 -`, if you pop in the wrong order, you get `3 - 5 = -2` instead of `5 - 3 = 2`.
- **Division truncation:** Most languages have different rules. Use `int(a / b)` in Python (truncates toward zero), and ensure C++ and Java do the same.
- **Negative numbers:** Tokens like `"-11"` are single tokens, not the operator `-` followed by `11`. Parse carefully.
- **Single operand:** The input `["42"]` is valid and should return 42 without any operations.
</Aside>

## Interview Tips

<Aside type="tip" title="Key trade-offs to discuss">
- **Stack vs Recursion:** Both are O(n) time and space. Stack is more common in production; recursion is more elegant for interviews if you want to demonstrate understanding of call stacks.
- **Why RPN?** RPN eliminates the need for parentheses and operator precedence rules. Many calculators use it internally.
- **Division edge case:** Clarify with the interviewer whether to truncate towards zero or use floor division. LeetCode uses truncation towards zero.
- **Performance note:** The stack approach avoids function call overhead, making it slightly faster in practice, though both are O(n).
- **Follow-ups to anticipate:**
  - "What if we wanted to support functions like `sin(x)` or `sqrt(x)`?" (Add special handling for function names)
  - "Can you convert infix to RPN?" (This is the Shunting Yard algorithm)
  - "What if operators had different precedence?" (Harder in RPN; easier in infix)
</Aside>
