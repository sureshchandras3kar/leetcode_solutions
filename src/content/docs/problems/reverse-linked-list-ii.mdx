## Problem Statement

Given the head of a linked list and two integers `left` and `right` where `left <= right`, **reverse the nodes of the list from position `left` to position `right`**, and return the reversed list.

<Aside type="tip" title="What to notice first">
- Positions are **1-indexed** (not 0-indexed), so position 1 is the head node.
- Only the sublist from position `left` to `right` is reversed; the rest stays in place.
- The reversal must be done **in-place** â€” do not allocate new nodes.
- A dummy node simplifies handling the case where reversal starts at the head.
</Aside>

### Examples

| Input | left | right | Output | Explanation |
| --- | --- | --- | --- | --- |
| `1â†’2â†’3â†’4â†’5â†’null` | 2 | 4 | `1â†’4â†’3â†’2â†’5â†’null` | Reverse nodes 2, 3, 4 |
| `5â†’null` | 1 | 1 | `5â†’null` | Single node, no reversal needed |
| `1â†’2â†’null` | 1 | 2 | `2â†’1â†’null` | Reverse the entire list |

## Constraints

- The number of nodes in the list is `n`.
- `1 <= n <= 500`
- `-500 <= Node.val <= 500`
- `1 <= left <= right <= n`

## Real-World Applications

<Aside type="note" title="Where this pattern appears">
- **Log rotation** â€” reverse a section of log entries in a linked structure.
- **Undo/redo stacks** â€” partially reverse command history from position m to n.
- **Playlist reordering** â€” reverse a range of songs in a linked playlist structure.
- **Data stream chunking** â€” reverse a window of elements in streaming linked data.
</Aside>

## Approach 1: Iterative with Pointers

<ApproachTag tag="recommended" />

Use a dummy node pointing to the head to simplify edge cases. Advance to position `left - 1` (the node before the reversal). Then, perform `right - left` reversals by moving nodes to the front of the sublist.

<ComplexityBadges time="O(n)" space="O(1)" timeColor="green" spaceColor="green" timeNote="Single pass through list" spaceNote="Pointer variables only" />

### Pseudocode

```plaintext
function reverseBetween(head, left, right):
    if left == right:
        return head
    
    dummy = ListNode(0, head)
    prev = dummy
    
    # Advance prev to position left-1
    for i in range(left - 1):
        prev = prev.next
    
    curr = prev.next
    
    # Reverse right - left times
    for i in range(right - left):
        next_temp = curr.next
        curr.next = next_temp.next
        next_temp.next = prev.next
        prev.next = next_temp
    
    return dummy.next
```

## Approach 2: Recursive

<ApproachTag tag="elegant" />

Recursively advance to position `left`. At that point, reverse nodes one at a time as the call stack unwinds, tracking when to stop at position `right`.

<ComplexityBadges time="O(n)" space="O(n)" timeColor="green" spaceColor="yellow" timeNote="Traverse to left, reverse to right" spaceNote="Recursion call stack" />

### Key Insight

Instead of explicit pointer swaps, recursion naturally tracks the "current position." When we reach position `left`, we begin the reversal. As we unwind (backtrack), we reverse links and stop after `right - left` reversals.

### Step-by-step Example

**Input:** `1â†’2â†’3â†’4â†’5â†’null`, left = 2, right = 4

| Step | Depth | Node | Action |
| --- | --- | --- | --- |
| Go deep | 1 | Node 1 | Recurse to node 2 |
| Go deep | 2 | Node 2 | Recurse to node 3 (mark as reversal start) |
| Go deep | 3 | Node 3 | Recurse to node 4 |
| Go deep | 4 | Node 4 | Recurse to node 5 |
| Unwind | 4 | Node 4â†’5 | Reverse: 4.next.next = 4 (stop here, right=4) |
| Unwind | 3 | Node 3â†’4 | Reverse: 3.next.next = 3 |
| Unwind | 2 | Node 2â†’3 | Reverse: 2.next.next = 2 (tail of reversed section) |
| Unwind | 1 | Node 1â†’2 | Connect tail; return |

### Pseudocode

```plaintext
function reverseBetween(head, left, right):
    if left == 1:
        return reverseN(head, right)
    else:
        head.next = reverseBetween(head.next, left - 1, right - 1)
        return head

function reverseN(head, n):
    if n == 1:
        return head
    
    new_head = reverseN(head.next, n - 1)
    head.next.next = head
    head.next = successor
    return new_head
```

<Aside type="caution" title="Watch for these pitfalls">
- **1-indexed positions:** Forget that left and right are 1-indexed, and your loops will be off by one.
- **Dummy node importance:** Without a dummy node, reversing from position 1 requires special handling; the dummy eliminates this edge case.
- **Tail connection:** Ensure the node before position `left` still points to the node before position `right+1` after reversal. The dummy and pointer tracking handle this.
- **Recursion stack depth:** For very large n and large reversal ranges, recursion may hit stack limits. The iterative approach is safer for production.
</Aside>

## Interview Tips

<Aside type="tip" title="Key trade-offs to discuss">
- **Why iterative?** It is O(1) space and avoids recursion depth limits â€” the safest production choice.
- **Why recursive?** It is more elegant and showcases understanding of recursion and backtracking â€” great for demonstrating depth.
- **Dummy node trick:** Mention this pattern applies to many linked list problems (removing nodes, reversing, etc.).
- **Edge cases:** Always test left=1 (reversal at head), right=n (reversal at tail), and left=right (no reversal).
- **Pointer safety:** Talk about null checks â€” ensure curr.next exists before accessing it.
