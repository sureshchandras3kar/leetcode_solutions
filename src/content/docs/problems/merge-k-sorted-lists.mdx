## Problem Statement

You are given an array of `k` linked-lists `lists`, each linked-list is sorted in ascending order.

Merge all the linked-lists into one sorted linked-list and return it.

<Aside type="tip" title="What to notice first">
- Compare elements across k lists efficiently; brute force is O(nk² log nk).
- Divide and conquer: merge pairs, then pairs of pairs → O(nk log k).
- Min-heap: always extract the smallest element from any list → O(nk log k).
- k lists of total n nodes merged pairwise is much faster than sequential pairwise merge.
</Aside>

### Examples

| Input | Output |
| --- | --- |
| `[[1,4,5],[1,3,4],[2,6]]` | `1→1→2→3→4→4→5→6` |
| `[]` | `null` |
| `[[]]` | `null` |

## Constraints

- `k == lists.length`
- `0 <= k <= 10^4`
- `0 <= lists[i].length <= 500`
- `-10^4 <= lists[i][j] <= 10^4`

## Real-World Applications

<Aside type="note" title="Where this pattern appears">
- Memory allocation and garbage collection
- LRU cache implementation
- Music playlist and browser history
- Undo/Redo functionality in applications
</Aside>

## Complexity Comparison

| Approach | Time | Space |
| --- | --- | --- |
| Recursive Divide & Conquer | O(nk log k) | O(log k) |
| Min Heap | O(nk log k) | O(k) |

*n = total nodes across all lists*

## Which approach should you learn first?

- Pick **Divide & Conquer** for pattern clarity — mirrors merge sort.
- Pick **Min Heap** for intuitive greedy selection of smallest.

## Approach 1: Recursive Divide and Conquer (Recommended)

<ApproachTag tag="recommended" />

1. Recursively split lists in half.
2. Merge each pair of resulting lists.
3. Base case: single list or empty range.

The key insight: merging k lists pairwise is O(log k) levels, each level does O(nk) work → O(nk log k) total.

## Approach 2: Min Heap

<ApproachTag tag="interview" />

1. Push all list heads into a min-heap.
2. Repeatedly extract the smallest, append to result, and push its next node.
3. Stop when heap is empty.

<ComplexityBadges time="O(nk log k)" space="O(k)" timeColor="green" spaceColor="yellow" timeNote="Each node processed once, heap ops" spaceNote="Heap storage" />

## Common mistakes

<Aside type="caution" title="Watch for these pitfalls">
- Comparing ListNode pointers instead of values (in heap, use custom comparator).
- Forgetting to push next node after extracting from heap.
- Off-by-one errors in partition boundaries.
- Not handling empty lists or k=0.
- Modifying list structure incorrectly during merge.
</Aside>

## Interview Tips

<Aside type="tip" title="Key trade-offs to discuss">
- **Divide & conquer vs. heap:** Divide & conquer uses recursion (stack space), heap uses explicit data structure. Both are O(nk log k).
- **Why not sequential merge?** Merging k lists one-by-one is O(nk²), which is slower.
- **Optimization:** Use iterative merge with size doubling instead of recursive partition for better cache locality.
- **Follow-up:** How would you stream the result without building the entire merged list in memory?
- **Edge cases:** Empty input, single list, all empty lists, lists of length 1.
</Aside>
